# The Local Tracing Proxy

**When:** 2026-03-25
**Context:** LangGraph migration era, after LangSmith tool call tracing was implemented

## Why the Proxy Exists

Early in the LangGraph migration the pipeline gained LangSmith observability: each work item
got a root `RunTree`, each agent call produced child spans, and the whole thing was visible
in the LangSmith dashboard. It was useful for about a week.

Then the free quota ran out.

Getting more LangSmith quota meant either paying or filing a request. Neither happened quickly.
More importantly, the LangSmith UI surfaced the wrong metrics. It showed runs and latencies
well, but there was no way to see "total cost per work item", "which item type is most
expensive", or "pipeline throughput over time" -- the numbers the pipeline actually needed to
optimize against. LangSmith is designed for debugging individual chains, not for tracking
the economics of an automated backlog processor.

So the pipeline built its own replacement.

## What the Proxy Does

`TracingProxy` (in `langgraph_pipeline/web/proxy.py`) is a lightweight shim that intercepts
the same HTTP calls the LangSmith SDK makes and stores them locally in a SQLite database
(`~/.claude/orchestrator-traces.db`). A custom web UI (`/proxy`) then surfaces exactly
the metrics that matter: cost per item, model breakdown, tool call attribution, and timeline
visualization.

The proxy activates automatically whenever the web server is running (`--web` flag). When
active, `configure_tracing()` sets `LANGCHAIN_ENDPOINT=http://localhost:{port}`, redirecting
all SDK traffic to the local server. No LangSmith credentials are needed; the API key is
set to `"local-proxy"` if no real key exists.

Forwarding to the real LangSmith API is still supported: set `forward_to_langsmith: true`
in `orchestrator-config.yaml` under `web.proxy`. The default is `false`, so no quota is
consumed unless explicitly opted in.

## Which LangSmith SDK Calls Are Intercepted

The server registers four LangSmith-compatible endpoints in `create_app()`:

| Endpoint | SDK action | Status |
|----------|-----------|--------|
| `GET /info` | SDK capability probe -- negotiates multipart vs. legacy mode | Intercepted |
| `POST /runs/multipart` | Batch ingestion (used by newer SDK versions and LangGraph's built-in tracer) | Intercepted |
| `POST /runs` | Single run creation (used by `RunTree.post()` in langsmith.py) | Intercepted |
| `PATCH /runs/{run_id}` | Run updates: `end_time`, `outputs`, `error` | Intercepted |

All four cover the complete SDK lifecycle: capability negotiation, run creation, child run
creation, and run finalization. There is no `GET /sessions` endpoint because the SDK does
not call one when `LANGCHAIN_ENDPOINT` is overridden -- session management is a cloud-UI
concept that does not exist in the on-wire API.

Because `LANGCHAIN_ENDPOINT` is set to the local server before the SDK makes any network
call, **all SDK traffic is redirected to the proxy**. Calls do not leak to
`api.smith.langchain.com`. The one edge case to watch for: if the `langsmith` package
performs any hardcoded-URL initialization at import time (outside the configured endpoint),
that call would bypass the proxy. In practice this has not been observed, and the API key
value `"local-proxy"` would cause authentication failures on the real LangSmith API anyway.

## Where Trace IDs Come From

Run UUIDs are generated by the LangSmith SDK. `RunTree` produces UUIDv7 identifiers
(time-ordered, e.g. `019d288e-61b7-7402-...`) when a new run is created. These IDs are
sent in the POST payload and stored verbatim in the local DB. The proxy never generates
its own IDs -- it only stores what the SDK sends.

This is currently fine because the SDK is still a dependency. If the SDK were ever removed
in favor of direct calls to the proxy, IDs would need to come from somewhere else. The
right choice would be UUIDv7 (via the `uuid7` PyPI package) to keep IDs time-sortable
without a separate `created_at` column -- the same format the current DB contains. UUIDv4
(random) would also work but would break chronological ordering by ID.

The decision to remove the SDK dependency is not on the table at this point. The SDK handles
the multipart encoding, run nesting, and context propagation that would need to be
reimplemented manually. The cost is worth the convenience.

## Architectural Summary

```
Pipeline worker
    |
    | (emit_tool_call_traces, create_root_run, finalize_root_run)
    |
LangSmith SDK (RunTree.post(), child.post())
    |
    | LANGCHAIN_ENDPOINT = http://localhost:{port}
    |
FastAPI server
    ├── GET  /info               (capability negotiation)
    ├── POST /runs/multipart     (batch ingest -- LangGraph tracer)
    ├── POST /runs               (single run -- RunTree.post())
    └── PATCH /runs/{run_id}    (run updates)
            |
            v
    TracingProxy.record_run()
            |
            v
    SQLite: ~/.claude/orchestrator-traces.db
            |
            v
    Web UI: /proxy (list), /proxy/{run_id} (detail)
```

No calls reach `api.smith.langchain.com` when the proxy is active and
`forward_to_langsmith` is `false`.
