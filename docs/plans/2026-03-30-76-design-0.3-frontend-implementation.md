# Design 0.3: Frontend Implementation for Markdown Rendering and Layout Fix

Source: tmp/plans/.claimed/76-item-page-markdown-rendering-and-layout.md
Requirements: docs/plans/2026-03-30-76-item-page-markdown-rendering-and-layout-requirements.md
Design overview: docs/plans/2026-03-30-76-item-page-markdown-rendering-and-layout-design.md

## Approach Summary

This design provides concrete, line-level frontend changes to item.html and a minimal
backend change to item.py. The guiding principle is minimal diff surface: reuse the
existing .requirements-body CSS class for rendered markdown, use the existing
_render_md_to_html() server function, and keep the dual-mode (pre vs div) logic in a
single shared helper function to avoid duplicating detection logic across two IIFEs.

---

## D1: Server-Side Endpoint Change (item.py)

### Current State (line 317-349)

The GET /item/{slug}/artifact-content endpoint accepts a path query parameter and
always returns PlainTextResponse.

### Proposed Change

Add an optional render query parameter. When render=html AND the file extension is .md,
use the existing _render_md_to_html() to return an HTMLResponse instead.

```python
@router.get("/item/{slug}/artifact-content")
def item_artifact_content(
    slug: str,
    path: str = Query(..., description="Relative path to the artifact file"),
    render: Optional[str] = Query(None, description="Set to 'html' for markdown rendering"),
) -> PlainTextResponse | HTMLResponse:
    # ... existing path validation (lines 337-348) unchanged ...

    # Markdown rendering: server-side HTML when requested
    if render == "html" and requested.suffix.lower() == ".md":
        html = _render_md_to_html(requested)
        if html:
            return HTMLResponse(content=html)

    return PlainTextResponse(
        content=requested.read_text(encoding="utf-8", errors="replace")
    )
```

Key decisions:
- Double-gate: both render=html AND .md extension must be true. This prevents
  accidental HTML injection for non-markdown files.
- Falls through to PlainTextResponse if _render_md_to_html() returns empty string
  (e.g. file is empty or markdown library fails).
- No new imports required -- HTMLResponse is already imported in item.py (line 31).

## D2: JavaScript Changes for Markdown-Aware Display

### Current State

Two independent IIFEs load artifact content:

1. **Step Explorer** (line 1535): loadStageArtifacts() iterates .step-artifact-content
   pre elements, fetches /artifact-content?path=..., sets pre.textContent.
2. **Output Artifacts** (line 1499): Click handler on .artifact-path-link buttons,
   fetches same endpoint, sets pre.textContent.

Both always treat content as plain text.

### Proposed Change: Shared Helper Function

Add a single top-level helper function (before both IIFEs) that encapsulates the
markdown detection and dual-mode display logic:

```javascript
/**
 * Fetch artifact content and display in the appropriate mode.
 * Markdown files (.md) are fetched with render=html and displayed as
 * rendered HTML in a div; all other files use plain text in a pre.
 *
 * @param {HTMLElement} container - The pre element (will be replaced by div for .md)
 * @param {string} artifactPath - Relative file path from data-artifact-path
 * @param {string} slug - Work item slug for URL construction
 */
function loadArtifactContent(container, artifactPath, slug) {
    var isMarkdown = /\.md$/i.test(artifactPath);
    var baseUrl = '/item/' + encodeURIComponent(slug) +
        '/artifact-content?path=' + encodeURIComponent(artifactPath);
    var url = isMarkdown ? baseUrl + '&render=html' : baseUrl;

    container.textContent = 'Loading\u2026';

    fetch(url)
        .then(function (r) {
            if (!r.ok) { throw new Error('HTTP ' + r.status); }
            return r.text();
        })
        .then(function (content) {
            if (isMarkdown) {
                // Replace pre with a styled div for rendered markdown
                var div = document.createElement('div');
                div.className = 'artifact-md-body requirements-body';
                div.innerHTML = content;
                container.parentNode.replaceChild(div, container);
            } else {
                container.textContent = content;
            }
        })
        .catch(function (err) {
            container.textContent = 'Failed to load: ' + err.message;
        });
}
```

### Why replaceChild Instead of innerHTML on the Pre

The pre element has specific CSS (monospace font, dark background, white-space: pre).
Rendered markdown needs entirely different styling (variable-width font, white
background, normal line wrapping). Replacing the element avoids CSS specificity
conflicts and keeps the DOM clean. The replacement div gets the .requirements-body
class which already has all the needed markdown styles.

### Step Explorer Update (loadStageArtifacts)

Replace the fetch/textContent logic inside loadStageArtifacts with a call to the
shared helper:

```javascript
function loadStageArtifacts(stageEl) {
    stageEl.querySelectorAll('.step-artifact-content').forEach(function (pre) {
        if (pre.dataset.loaded) { return; }
        pre.hidden = false;
        pre.dataset.loaded = '1';
        loadArtifactContent(pre, pre.dataset.artifactPath, slug);
    });
}
```

Lines changed: 1536-1550 (replace 15 lines with 7).

### Output Artifacts Click Handler Update

Replace the fetch/textContent logic in the click handler:

```javascript
btn.addEventListener('click', function () {
    var entry = btn.closest('.artifact-entry');
    var viewer = entry.querySelector('.artifact-viewer');
    var pre = viewer.querySelector('pre');
    var expanded = btn.getAttribute('aria-expanded') === 'true';

    if (expanded) {
        viewer.hidden = true;
        btn.setAttribute('aria-expanded', 'false');
        return;
    }

    viewer.hidden = false;
    btn.setAttribute('aria-expanded', 'true');

    if (pre.dataset.loaded) { return; }
    pre.dataset.loaded = '1';
    loadArtifactContent(pre, btn.dataset.artifactPath, slug);
});
```

Lines changed: 1515-1525 (replace 11 lines with 3 effective lines of fetch logic).

### Edge Case: Already-Loaded Check After Replacement

When a markdown artifact has been loaded (pre replaced by div), the data-loaded
attribute lives on the original pre which is now detached. The guard check at
"if (pre.dataset.loaded) { return; }" uses the pre queried fresh from the DOM.
After replacement, the pre no longer exists in the DOM, so querySelector('pre')
returns null. This needs a guard:

```javascript
// In the output artifacts click handler:
var contentEl = viewer.querySelector('pre') || viewer.querySelector('.artifact-md-body');
if (!contentEl || contentEl.dataset.loaded) { return; }
```

For the step explorer, the same issue applies. The querySelectorAll selector should
be broadened:

```javascript
stageEl.querySelectorAll('.step-artifact-content, .artifact-md-body').forEach(...)
```

But since .artifact-md-body elements will always have been loaded (they only exist
after replacement), the dataset.loaded check handles this correctly -- no re-fetch.

A cleaner alternative: keep the data-loaded attribute on the parent container instead
of the pre/div element. For the step explorer, the parent is .step-artifact; for
output artifacts, the parent is .artifact-viewer. This decouples the loaded state
from the display element:

```javascript
// Step explorer: mark parent as loaded
var artifactEl = pre.closest('.step-artifact');
if (artifactEl.dataset.loaded) { return; }
artifactEl.dataset.loaded = '1';
loadArtifactContent(pre, pre.dataset.artifactPath, slug);
```

This approach is recommended because it survives element replacement without any
selector gymnastics.

## D3: CSS Grid Overflow Containment

### Current State (lines 195-204)

```css
.item-layout {
    display: grid;
    grid-template-columns: 1fr;
    gap: 1.5rem;
    align-items: start;
}

@media (min-width: 900px) {
    .item-layout { grid-template-columns: 1fr 360px; }
}
```

The left column grid child (first div inside .item-layout, line 1144) has no
min-width constraint. CSS grid items default to min-width: auto, which means
they cannot shrink below their content width. Wide pre content (long lines in
artifacts) forces the left column wider than 1fr, pushing the 360px sidebar
off-screen or under a horizontal scrollbar.

### Proposed CSS Change

Add min-width: 0 and overflow: hidden to the left column grid child:

```css
/* Grid children must allow shrinking below content width (D3) */
.item-layout > :first-child {
    min-width: 0;
    overflow: hidden;
}
```

Place this rule immediately after the .item-layout media query block (after line 204).

### Why min-width: 0 AND overflow: hidden

- min-width: 0 allows the grid algorithm to allocate the column correctly. Without it,
  the grid resolves 1fr as "at least as wide as the widest content."
- overflow: hidden clips any content that still escapes after grid sizing. This is a
  safety net -- in practice min-width: 0 is sufficient for most content, but extremely
  wide inline elements (e.g. a very long URL without word-break) could still overflow
  the grid cell without it.
- overflow: hidden is chosen over overflow: auto because the scrolling should happen
  at the artifact-content level (each pre/div has its own overflow-x: auto), not at
  the column level. Column-level scrollbars would be confusing UX.

### Why :first-child Instead of a Named Class

The template structure is:
```html
<div class="item-layout">
  <div>  <!-- left column: step explorer, requirements, etc. -->
  <div>  <!-- right column: plan tasks, completions, artifacts -->
</div>
```

Neither div has a dedicated class. Adding a class (e.g. .item-layout-left) would be
cleaner long-term but is outside the minimal-change scope. Using :first-child is
unambiguous given the fixed two-child structure and avoids template changes.

## D4: Rendered Markdown Styles Within Artifact Viewers

### Current State

The .requirements-body class (lines 242-326) provides comprehensive markdown styling:
headings (h1-h4), paragraphs, lists, inline code, fenced code blocks (pre > code),
tables, and blockquotes. It assumes a white background with dark text.

The .step-artifact-content (lines 1010-1023) and .artifact-viewer-pre (lines 857-868)
use a dark background (#1a1a2e) with light text (#e2e8f0) -- these are for raw text
display.

### Proposed Approach: Dual-Class With Override

The shared loadArtifactContent function creates a div with class
"artifact-md-body requirements-body". The .artifact-md-body class provides the
container context (padding, max-height, overflow), while .requirements-body provides
the typographic styling.

```css
/* ── Rendered markdown in artifact viewers (D4) ─────────────────────── */
.artifact-md-body {
    padding: 1rem 1.25rem;
    max-height: 400px;
    overflow-y: auto;
    overflow-x: auto;
    background: #fff;
    border: 1px solid #e2e8f0;
    border-radius: 6px;
    margin: 0.4rem 0 0;
}

/* Ensure code blocks inside rendered markdown scroll horizontally */
.artifact-md-body pre {
    max-width: 100%;
    overflow-x: auto;
}
```

### Why White Background for Rendered Markdown

The existing .requirements-body styles assume dark-on-light color scheme (color: #374151,
heading color: #1a1a2e, code background: #f1f5f9). Rendering markdown on the dark
artifact background (#1a1a2e) would require duplicating every color rule with inverted
values. Using a white background lets us reuse .requirements-body as-is.

This creates a visual distinction between raw text artifacts (dark background) and
rendered markdown (white background), which actually improves UX by making the
rendering mode immediately obvious.

### For the Step Explorer Context

The step explorer artifacts sit inside .step-stage-body which has no background.
When a step artifact pre gets replaced by a div.artifact-md-body, the white
background and border provide clear visual boundaries. The margin-top: 0.4rem
matches the existing .step-artifact-content margin.

### For the Output Artifacts Context

The output artifact viewer (.artifact-viewer) already has overflow: hidden and a
border. When the pre inside gets replaced by a div.artifact-md-body, the outer
border from .artifact-viewer will stack with the inner border from .artifact-md-body.
To avoid double borders:

```css
.artifact-viewer .artifact-md-body {
    border: none;
    border-radius: 0;
    margin: 0;
}
```

This lets the parent .artifact-viewer provide the border and border-radius, while the
.artifact-md-body provides padding and overflow behavior.

## D5: Template Changes in item.html

### Step Explorer Template (lines 1178-1180)

No template change needed. The pre element is created in Jinja2:
```html
<pre class="step-artifact-content"
     data-artifact-path="{{ artifact.path }}"
     hidden>Loading&hellip;</pre>
```

JavaScript replaces this pre with a div when the artifact is markdown. The template
continues to emit pre for all artifacts; the runtime JS handles the swap.

### Output Artifacts Template (lines 1277-1279)

Similarly, no template change needed:
```html
<div class="artifact-viewer" hidden aria-live="polite">
    <pre class="artifact-viewer-pre">Loading&hellip;</pre>
</div>
```

JavaScript replaces the pre when content is markdown.

### Why No Template Changes

Server-side template changes (e.g. emitting different elements for .md files)
would require the backend to pass markdown detection data to the template context,
and would bifurcate the Jinja2 template with if/else blocks. The JavaScript approach
is cleaner because:

1. The endpoint already knows the file extension (it resolves the path).
2. The JS helper already has the path string for extension detection.
3. Element replacement is a single DOM operation with no visual flash.
4. It avoids coupling the template to file-type knowledge.
