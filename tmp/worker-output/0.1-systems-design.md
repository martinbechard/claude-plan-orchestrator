## 1. Architecture Overview

The execution history UI is a three-layer architecture bridging SQLite trace data
to an interactive tree view with detail panels:

```
+-------------------+     +---------------------+     +---------------------------+
|  Data Layer       | --> |  View Model Layer   | --> |  Presentation Layer       |
|  (TracingProxy)   |     |  (trace_narrative)  |     |  (Templates + JS)         |
|                   |     |                     |     |                           |
|  - get_full_      |     |  - build_execution_ |     |  - execution_history.html |
|    subtree()      |     |    tree()           |     |  - execution_history.js   |
|  - get_node_      |     |  - TreeNode model   |     |  - completions.html       |
|    detail()       |     |  - NodeDetail model |     |    (trace link column)    |
|  - recursive CTE  |     |  - NodeType enum    |     |  - base.html (nav update) |
|    time spans     |     |  - MetricsView      |     |                           |
+-------------------+     +---------------------+     +---------------------------+
```

### Key Architectural Decisions

1. **Server-rendered tree with JS expand/collapse**: The initial tree (top 2 levels)
   is server-rendered in the Jinja2 template. Deeper levels are fetched lazily via
   JSON API endpoints when the user expands a node. This balances first-paint speed
   with unlimited depth navigation.

2. **Three dedicated API endpoints** instead of one monolithic endpoint:
   - Tree fetch (full subtree as flat rows)
   - Node detail (single node with parsed inputs/outputs/metadata)
   - Children fetch (lazy-load children of a specific node)

3. **Unified TreeNode model** replaces the current PhaseView/ExecutionView split.
   Every level of the hierarchy uses the same data structure, enabling recursive
   rendering without depth-specific code.

## 2. Data Models

### 2.1 TreeNode (View Model)

Replaces both ExecutionView and PhaseView with a single recursive structure:

```python
from dataclasses import dataclass, field
from enum import Enum
from typing import Optional


class NodeType(Enum):
    """Classification of a tree node by its role in the pipeline."""
    PIPELINE_ROOT = "pipeline_root"   # Top-level LangGraph run
    GRAPH_NODE = "graph_node"         # intake, execute_plan, validate, archive
    SUBGRAPH_NODE = "subgraph_node"   # executor subgraph (task_selection, task_running)
    AGENT_SESSION = "agent_session"   # Claude CLI session (coder, validator, etc.)
    TOOL_CALL = "tool_call"           # Read, Edit, Write, Bash, Grep, Glob, etc.
    UNKNOWN = "unknown"


@dataclass
class MetricsView:
    """Conditional metrics -- fields are None when data is absent (FR6: AC42)."""
    latency_ms: Optional[float] = None       # AC38
    latency_display: Optional[str] = None    # Human-readable "1.23s" or "2m 05s"
    input_tokens: Optional[int] = None       # AC39
    output_tokens: Optional[int] = None      # AC39
    cost_usd: Optional[float] = None         # AC40
    cost_display: Optional[str] = None       # "$0.0123"
    model: Optional[str] = None              # AC41


@dataclass
class ObservabilityData:
    """Structured observability metadata extracted from metadata_json (FR7)."""
    validator_verdict: Optional[str] = None       # AC43: "PASS"/"FAIL"/"WARN"
    validator_findings: Optional[list[str]] = None # AC43: list of finding strings
    pipeline_decision: Optional[str] = None        # AC44: "skip"/"retry"/"proceed"
    subprocess_exit_code: Optional[int] = None     # AC45
    plan_state_snapshot: Optional[dict] = None     # AC46: partial plan YAML state


@dataclass
class TreeNode:
    """Unified recursive tree node for the execution history view.

    Every node -- from pipeline root to leaf tool call -- uses this same structure.
    The node_type field determines which detail panel variant to render (UC3).
    """
    run_id: str
    parent_run_id: Optional[str]
    name: str                              # Resolved display name (never "LangGraph")
    node_type: NodeType
    status: str                            # "success" | "error" | "running" | "unknown"
    metrics: MetricsView
    has_children: bool                     # Drives expand/collapse affordance in UI
    children: list["TreeNode"] = field(default_factory=list)  # Populated lazily
    depth: int = 0                         # 0 = root

    # Phase classification for graph nodes
    phase_label: Optional[str] = None      # "Intake", "Execution", etc.

    # Activity summary (aggregated from descendants)
    activity_summary: Optional[str] = None # "Read 5 files, edited 2, ran 8 bash"

    # Observability data (FR7)
    observability: Optional[ObservabilityData] = None
```

## 4. API Boundaries

### 4.1 Endpoint: GET /proxy/{run_id}/execution (HTML)

Primary entry point from Completions page trace links. Returns server-rendered
execution_history.html with the tree pre-expanded to depth 2.

**Request:**
```
GET /proxy/{run_id}/execution
```

**Processing:**
1. TracingProxy.get_run(run_id) -- fetch root run
2. TracingProxy.get_full_subtree(run_id) -- recursive CTE fetch of all descendants
3. build_execution_tree(root_run, flat_descendants) -- assemble TreeNode hierarchy
4. Trim tree to depth 2 for initial HTML render (deeper nodes loaded lazily)
5. Render execution_history.html with tree + header metrics

**Response:** HTML page with embedded initial tree JSON for JS hydration.

**AC Coverage:** AC6, AC7, AC8 (completions entry point), AC9-AC11 (tree structure)

## 5. Data Flow

### 5.1 Full Subtree Fetch (New TracingProxy Method)

```sql
-- get_full_subtree(root_run_id): Recursive CTE with no depth limit
WITH RECURSIVE subtree AS (
    SELECT run_id, parent_run_id, name, start_time, end_time,
           inputs_json, outputs_json, metadata_json, error, model, created_at,
           0 AS depth
    FROM traces
    WHERE run_id = :root_run_id

    UNION ALL

    SELECT t.run_id, t.parent_run_id, t.name, t.start_time, t.end_time,
           t.inputs_json, t.outputs_json, t.metadata_json, t.error, t.model,
           t.created_at,
           s.depth + 1
    FROM traces t
    INNER JOIN subtree s ON t.parent_run_id = s.run_id
)
SELECT * FROM subtree
ORDER BY depth ASC, start_time ASC
```

Returns flat rows with parent_run_id linkage + depth. Python assembles into tree.

### 5.2 Tree Assembly (View Model Layer)

```python
def build_execution_tree(
    root_row: dict,
    descendant_rows: list[dict],
    cost_map: dict[str, float],      # From get_child_costs_batch
    time_span_map: dict[str, ChildTimeSpan],  # From recursive time spans
) -> ExecutionTreeResponse:
    """Assemble flat DB rows into a recursive TreeNode hierarchy.

    Algorithm:
    1. Deduplicate rows by run_id (keep row with end_time, FR5/AC36-AC37)
    2. Build lookup: run_id -> row
    3. Build children lookup: parent_run_id -> [child rows]
    4. Recursive construction from root:
       a. Classify node_type
       b. Resolve display name (slug resolution tiers, never "LangGraph")
       c. Compute metrics from metadata + cost_map + time_span_map
       d. Extract observability data
       e. Determine has_children from children lookup
       f. Recurse into children
    5. Compute header metrics (total cost, total duration, overall status)
    """
```

### 5.3 Node Type Classification

```python
def classify_node_type(name: str, metadata: dict, depth: int) -> NodeType:
    """Classify a trace row into a NodeType based on its characteristics.

    Classification rules (checked in order):
    1. depth == 0 -> PIPELINE_ROOT
    2. name matches a known tool name (Read, Edit, Bash, etc.) -> TOOL_CALL
    3. metadata contains 'agent_type' or 'agent' -> AGENT_SESSION
    4. name contains subgraph indicators (task_selection, task_running) -> SUBGRAPH_NODE
    5. name matches phase patterns (intake, execute, validate, etc.) -> GRAPH_NODE
    6. fallback -> UNKNOWN
    """
```

The classification is deterministic and stateless -- it depends only on the row
data, not on neighboring rows. This enables classification during tree assembly
without extra DB queries.

### 5.4 Slug Resolution Chain

Applied to the root node display name. Multi-tier with "LangGraph" guard:

```python
def resolve_display_name(row: dict, children_metadata: list[dict]) -> str:
    """Resolve the display name, never returning 'LangGraph'.

    Tier 1: metadata_json -> item_slug (AC32)
    Tier 2: metadata_json -> slug
    Tier 3: Child span metadata lookup for item_slug (AC33)
    Tier 4: run_id prefix (first 8 chars) as last resort (AC34)

    Guard: If resolved name == "LangGraph", advance to next tier (AC35)
    """
```

### 5.5 Metrics Computation

```python
def compute_metrics(

    cost_map: dict[str, float],
    time_span_map: dict[str, ChildTimeSpan],

    Latency:
    - Primary: end_time - start_time from the row
    - Fallback: recursive time_span_map when row duration < _NEAR_ZERO_DURATION_S

    - From metadata_json total_cost_usd for leaf nodes
    - Filtered: 0.0 treated as absent (AC40, AC4)

    Model:
    - From metadata_json or model column (AC41)
    - None when absent (AC42)
    """
```

## 9. Deduplication Strategy (D5 / AC36-AC37)

During tree assembly, before building the children lookup:

```python
def deduplicate_rows(rows: list[dict]) -> list[dict]:
    """Keep only one row per run_id.

    When both a START and END event exist for the same run_id:
    - Keep the row with end_time (more complete)
    - If both have end_time, keep the one with later created_at

    This is a UI-layer safety net atop the DB INSERT OR REPLACE.
    """
    by_run_id: dict[str, dict] = {}
    for row in rows:
        rid = row["run_id"]
        if rid not in by_run_id:
            by_run_id[rid] = row
        else:
            existing = by_run_id[rid]
            # Prefer row with end_time
            if row.get("end_time") and not existing.get("end_time"):
                by_run_id[rid] = row
            elif row.get("end_time") and existing.get("end_time"):
                if (row.get("created_at") or "") > (existing.get("created_at") or ""):
                    by_run_id[rid] = row
    return list(by_run_id.values())
```

## 10. Performance Considerations

### 10.1 Lazy Loading

The full subtree CTE can return hundreds or thousands of rows for a complex
pipeline run. To keep the initial page load fast:

- Server renders depth 0-1 (root + graph nodes) inline in HTML
- Depth 2+ nodes are included in a JSON blob embedded in a script tag
- JS expands depth 2 automatically on page load
- Depth 3+ children are fetched lazily via /api/proxy/{run_id}/children

### 10.2 Batch Queries

The existing batch query pattern (get_child_costs_batch, get_child_time_spans_batch)
is preserved for enriching the initial tree. A single call enriches all visible nodes
at once rather than N+1 individual queries.

### 10.3 Cost Aggregation Optimization

The get_child_costs_batch recursive CTE already traverses the full subtree. For the
execution history page, this CTE is called once for all visible nodes, and the results
are cached in the cost_map dict passed to build_execution_tree().

Zero-cost filtering: rows where total_cost_usd is 0.0 or NULL are excluded from
aggregation sums, ensuring placeholder costs never inflate displayed values (AC4, AC28).
