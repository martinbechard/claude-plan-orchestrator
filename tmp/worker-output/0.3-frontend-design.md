## 1. Problem Statement

The plan task lifecycle is gaining a sixth status ("verified") that separates
execution completion from validation completion. The frontend must visually
distinguish all six states -- especially "completed" (awaiting validation) from
"verified" (fully done) -- across three rendering surfaces:

1. **Plan task list** (item.html) -- individual task rows with status icons
2. **Progress counter** (item.html header) -- "X / Y" done count
3. **Progress bar** (new) -- colored segments showing state distribution

The existing dashboard (dashboard.html, dashboard.js, style.css) renders
pipeline-level worker activity, not plan task statuses. Plan task rendering lives
entirely in item.html via inline styles, Jinja2 template conditionals, and an
inline JS block for dynamic refresh.

## 2. Six-State Color Palette

### 2.1 Design Rationale

The current "completed" status is green (#d1fae5), which conveys "done." Under
the new lifecycle, "completed" means "awaiting validation" -- an intermediate
state that should NOT convey finality. The green must move to "verified," and
"completed" must shift to an amber/blue tone that says "in progress."

Color choices are driven by universal semantics and WCAG 2.1 AA contrast:

| Status | Background | Foreground | Icon | Semantic | WCAG Ratio |
|---|---|---|---|---|---|
| pending | #f3f4f6 | #d1d5db | (empty) | Not started, neutral | 1.4:1 (decorative) |
| in_progress | #dbeafe | #1e40af | &#9679; (filled circle) | Active work, blue | 8.6:1 |
| completed | #fef3c7 | #92400e | &#9201; (timer) | Awaiting validation, amber | 7.5:1 |
| verified | #d1fae5 | #065f46 | &#10003; (checkmark) | Fully done, green | 7.1:1 |
| failed | #fee2e2 | #991b1b | &#10007; (cross) | Error, red | 7.8:1 |
| skipped | #e5e7eb | #9ca3af | &mdash; (dash) | Skipped, grey | 2.8:1 (decorative) |

### 2.2 Accessibility Considerations

- **Color-blind safe**: The palette avoids relying solely on red/green
  distinction. "completed" uses amber (warm yellow), "verified" uses green, and
  "failed" uses red. Each pair is distinguishable under protanopia, deuteranopia,
  and tritanopia because they differ in both hue and luminance.
- **Icons as secondary signal**: Every status has a unique icon glyph, so color
  is never the sole differentiator (WCAG 1.4.1 Use of Color).
- **Status text on hover/focus**: The title attribute and aria-label spell out
  the status name, providing a text-based signal independent of visual styling.
- **High contrast**: All foreground-on-background combinations exceed 4.5:1
  contrast ratio for the status text labels, meeting WCAG AA for small text.
  The pending and skipped icons use lower contrast deliberately since they are
  decorative indicators, not informational text.

### 2.3 Key Semantic Shift

| Before | After |
|---|---|
| completed = green checkmark, strikethrough text (signals "done") | completed = amber timer icon (signals "awaiting next step") |
| (no verified state) | verified = green checkmark, strikethrough text (signals "fully done") |
| (no failed state for tasks) | failed = red cross (signals "error") |

The amber color for "completed" deliberately echoes the existing outcome-warn
badge (#fef3c7 / #92400e) used elsewhere in the dashboard, creating a visual
link to the concept of "needs attention / not finished."

### 5.3 Progress Bar Rendering (New)

Add a progressBarHtml function for the dynamic refresh path:

```javascript
function progressBarHtml(tasks) {
    if (!tasks || tasks.length === 0) return '';
    var total = tasks.length;
    var counts = { verified: 0, completed: 0, in_progress: 0, failed: 0, skipped: 0 };
    for (var i = 0; i < total; i++) {
        var s = tasks[i].status;
        if (counts.hasOwnProperty(s)) counts[s]++;
    }

    var segs = '';
    var order = ['verified', 'completed', 'in_progress', 'failed', 'skipped'];
    var titles = {
        verified: ' verified',
        completed: ' awaiting validation',
        in_progress: ' in progress',
        failed: ' failed',
        skipped: ' skipped'
    };
    for (var j = 0; j < order.length; j++) {
        var key = order[j];
        if (counts[key] > 0) {
            var pct = (counts[key] / total * 100).toFixed(1);
            segs += '<div class="progress-seg progress-seg--' + key +
                '" style="width:' + pct + '%"' +
                ' title="' + counts[key] + titles[key] + '"></div>';
        }
    }

    return '<div class="progress-bar" role="progressbar"' +
        ' aria-valuenow="' + counts.verified + '"' +
        ' aria-valuemin="0" aria-valuemax="' + total + '"' +
        ' aria-label="Task progress: ' + counts.verified +
        ' of ' + total + ' verified">' + segs + '</div>';
}
```

## 6. Backend Data Flow

### 6.1 _load_plan_tasks (routes/item.py)

The existing function (lines 509-554) already extracts task status from the
YAML plan. No changes are needed -- it reads whatever status value the YAML
contains, including the new "verified" and "failed" values.

### 6.2 effective_status Integration

The task list in the frontend receives raw task status values from the YAML plan.
The backend effective_status() helper (from D2) is used in the executor for
dependency resolution and progress counting, but the frontend should display the
raw stored status to give the user an accurate view of the actual YAML state.

This means:
- A legacy task with status "completed" (no validation configured) will show as
  amber "awaiting validation" in the UI, which is technically correct since the
  backend uses effective_status to treat it as verified for dependency purposes.
- Once the validator node advances the status to "verified" in the YAML, the UI
  updates to green.

There is no need for the frontend to implement effective_status logic. The raw
status display is correct and informative.

## 10. Backward Compatibility

### 10.1 Plans Without "verified" Status

Existing plans that use "completed" as the terminal state will display all tasks
as amber "awaiting validation" in the UI. This is visually different from the
current green, but semantically correct: those tasks have not been through the
new validation flow.

The backend effective_status() ensures these tasks are treated as verified for
dependency resolution purposes. The frontend does not need to replicate this
logic -- it shows the raw YAML status, which is the accurate ground truth.

### 10.2 CSS Class Convention

The CSS class pattern task-icon--{{ status }} is already used with raw status
values injected via Jinja2 template variables. Adding new status values to the
TaskStatus type automatically generates valid CSS class names without any mapping
layer. This convention is preserved for the new "verified" and "failed" statuses.

## 11. Design Principles

1. **Single source of truth**: The YAML status value drives everything -- CSS
   class, icon, text treatment, and progress counting. No translation layer.
2. **Amber = waiting, green = done**: The strongest visual signal is the
   color shift. Users learn quickly that amber means "not finished yet."
3. **Icons as redundant signal**: Each status has a unique icon, so the design
   works for color-blind users and in grayscale print.
4. **Minimal change surface**: All modifications are in item.html. No new files,
   no changes to the pipeline dashboard, no API changes.
5. **Progressive disclosure**: The progress bar gives a quick overview; the
   task list provides detailed per-task status; title/aria-label provides text
   labels for accessibility tools.
