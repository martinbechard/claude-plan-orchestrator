## 1. Design Philosophy

The item detail page transforms from a flat two-column layout into a vertical
stage-based step explorer. Users currently face a long scrolling page of
unrelated cards (Raw Input, Clause Register, 5 Whys, Structured Requirements,
Design, Plan Tasks, Output Artifacts, Validation Results, Completion History,
Traces). This makes it hard to answer the question "What happened at each stage
of the pipeline?"

The step explorer groups artifacts under their pipeline stage, creating a
mental model that mirrors how the pipeline actually processes work items.

Three principles:

1. **Progressive disclosure** -- Only the active or most recently completed
   stage is expanded on first visit. All other stages are collapsed to a
   single-line header showing stage name, status badge, and completion
   timestamp. Users drill into exactly the stage they care about.
2. **Temporal narrative** -- Stages are ordered chronologically (Intake first,
   Archive last). Within each stage, artifacts are ordered by creation time.
   Timestamps are visible on both stage headers and individual artifacts,
   making it easy to reconstruct the timeline of the pipeline run.
3. **Zero-cost browsing** -- Artifact content is not loaded until the user
   explicitly expands a stage. A JS Map cache ensures re-expanding a stage
   does not re-fetch content. The initial page load transfers only stage
   metadata (names, statuses, timestamps, artifact file paths).

## 4. Stage Body (Expanded Content)

### 4.1 Artifact List Layout

When a stage is expanded, its body shows a list of artifacts. Each artifact
is a row with:

```
[Artifact Icon] [Artifact Name]    [Timestamp]    [Size]    [View button]
```

Example for the Intake stage expanded:

```
 1  Intake                           done  Mar 28, 14:32    ^
 ───────────────────────────────────────────────────────────────
 |  User Request            Mar 28, 14:30    1.2 KB    [View]
 |  Clause Register         Mar 28, 14:31    0.8 KB    [View]
 |  5 Whys Analysis         Mar 28, 14:32    1.5 KB    [View]
```

### 4.2 Artifact Row Styling

- Font size: 13px (smaller than stage header's 14px)
- Left padding: 40px (aligns with stage name, past the step number circle)
- Border-left: 2px solid #e2e8f0 (connecting line from header to artifacts)
- Each row has a thin bottom separator (#f0f4f8)
- The connecting left border creates a visual "tree" structure

### 4.3 Artifact Content Viewer

Clicking "View" on an artifact toggles an inline content viewer below the
artifact row. This reuses the existing artifact-viewer pattern from the
current Output Artifacts card:

```
 |  User Request            Mar 28, 14:30    1.2 KB    [Hide]
 |  ┌──────────────────────────────────────────────────────┐
 |  │  # Feature: Step Explorer                            │
 |  │                                                      │
 |  │  As a user, I want to see pipeline artifacts         │
 |  │  grouped by stage so I can understand the            │
 |  │  processing timeline...                              │
 |  └──────────────────────────────────────────────────────┘
```

The viewer container:
- Background: #1a1a2e (dark, matching existing code viewer)
- Border-radius: 6px
- Max-height: 400px with overflow-y: auto
- Monospace font, 11px, color #e2e8f0
- Margin: 4px 0 8px 40px (aligned with artifact text)

### 4.4 Loading State

When artifact content is being fetched:

```
 |  User Request            Mar 28, 14:30    1.2 KB    [View]
 |  ┌──────────────────────────────────────────────────────┐
 |  │  Loading...                                          │
 |  │  ▓▓▓▓▓▓▓▓▓▓░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░       │
 |  └──────────────────────────────────────────────────────┘
```

Loading indicator design:
- Text "Loading..." in muted color (#6b7280, 12px)
- Below it, a CSS-only animated progress bar (indeterminate)
- Progress bar: 4px height, #3b6fcf moving gradient on #e2e8f0 background
- Animation: a 1.5s ease-in-out sliding gradient
- Replaces with actual content once fetch completes
- If fetch fails: "Failed to load content" in #991b1b, with a "Retry" link


## 5. Collapse/Expand Interaction Patterns

### 5.1 Toggle Behavior

**Click anywhere on the stage header** to toggle expand/collapse. The entire
header row is the clickable target, not just the chevron. This provides a
large hit target and matches the mental model of "clicking a section to open it."

**Chevron rotation:** The chevron (right-pointing triangle) rotates 90 degrees
clockwise when expanding. CSS transition: `transform 150ms ease`.

**Body reveal:** The stage body slides open using CSS `max-height` transition
(200ms ease) from 0 to the content height. For stages with many artifacts,
the max-height is computed dynamically by JavaScript to avoid the common
CSS max-height animation pitfall of using an arbitrary large value.

### 5.2 Independent Stage State

Each stage opens and closes independently. Opening one stage does NOT close
others. This is a multi-expand accordion, not a single-panel accordion.

Rationale: Users often want to compare artifacts across stages (e.g., checking
if the design document addresses all structured requirements). Forcing
single-panel behavior would require constant re-expanding.

### 5.3 Default Expand Behavior (First Visit)

On the first visit to an item page (no localStorage entry for this slug):

1. Find the **most recent active stage** -- the first stage (in pipeline
   order) whose status is "in_progress". If none, use the last "done" stage.
2. Expand that stage. Collapse all others.
3. If all stages are "not_started" (e.g., newly queued item), expand the
   Intake stage.

This ensures the user immediately sees the most relevant content without
needing to hunt for the active stage.

### 5.4 Persistence via localStorage

After the first visit, collapse/expand state is persisted in localStorage.
The key format is:

```
step-explorer:{slug}:{stage_id}
```

Values: `"open"` or `"closed"`.

On subsequent visits:
1. Read localStorage for each stage
2. If a key exists, restore that state
3. If no key exists for a stage (e.g., a new stage appeared), use the default
   behavior from 5.3 for that stage only

### 5.5 Expand All / Collapse All

A small control pair in the step explorer header area:

```
Step Explorer                    [Expand All] [Collapse All]
```

- Font size: 11px
- Style: text links, not buttons (#3b6fcf, underline on hover)
- "Expand All" sets all stages to open and updates localStorage
- "Collapse All" sets all stages to closed and updates localStorage
- Keyboard shortcut: none (keep it simple)

### 5.6 Not-Started Stages

Stages with status "not_started" and zero artifacts have no content to show.
These stages:
- Are NOT clickable (no pointer cursor, no hover effect)
- Show the stage header in a muted style (opacity: 0.6)
- No chevron icon (nothing to expand)
- Header still shows the step number, name, and "Not Started" badge

This prevents user confusion from clicking a stage and seeing an empty body.

## 6. Timestamp Design

### 6.1 Stage Completion Timestamps

Shown only when `status == "done"`. Positioned right-aligned in the stage
header row, before the chevron.

Format: **Relative time** for recent timestamps, **absolute date-time** for
older ones:

| Age | Format | Example |
|---|---|---|
| < 1 minute | "Just now" | Just now |
| 1-59 minutes | "Xm ago" | 12m ago |
| 1-23 hours | "Xh ago" | 3h ago |
| 1-6 days | "Xd ago" | 2d ago |
| > 6 days | "MMM DD, HH:MM" | Mar 28, 14:32 |

The timestamp is rendered in a `<time>` element with the ISO datetime in the
`datetime` attribute for machine readability. The human-friendly text is the
element content.

### 6.2 Artifact Timestamps

Each artifact row shows its file modification time. Same format rules as
stage timestamps. Positioned after the artifact name, before the file size.

### 6.3 Timestamp Rendering

Timestamps are rendered server-side by Jinja2. The relative-time computation
happens in Python (item.py) at request time. For live-updating pages, the
`/dynamic` endpoint returns raw ISO timestamps, and JavaScript computes the
relative format client-side (using the same rules) to keep the display fresh
during polling.

A `<time>` element with `class="relative-time"` and `datetime="{{ iso }}"` is
used. The JS polling update function can refresh these by recomputing the
relative text from the datetime attribute.

## 7. Live Updates Integration

### 7.1 Stage Status Updates via /dynamic

The existing `/dynamic` polling endpoint is extended to include stage data.
When the polling JS receives a response, it updates:

- Stage status badges (not_started -> in_progress -> done)
- Stage completion timestamps (appear when a stage transitions to done)
- New artifacts (added to the artifact list when discovered)
- Active stage indicator (pulse animation moves to the new active stage)

### 7.2 New Artifact Discovery

When `/dynamic` returns a stage with more artifacts than currently shown:
- New artifacts are appended to the artifact list in the correct position
- A subtle slide-in animation (translateY from -8px, opacity 0 to 1, 200ms)
  indicates the new arrival
- The content cache (JS Map) does not have an entry for the new artifact,
  so it will be fetched on first view

### 7.3 Stage Transition Animation

When a stage transitions from "in_progress" to "done":
1. Status badge animates: brief background pulse (green flash, 400ms)
2. Completion timestamp fades in (opacity 0 to 1, 300ms)
3. The pulsing dot on the old badge stops

When the next stage transitions to "in_progress":
1. New status badge appears with the pulsing dot
2. If the stage was collapsed and has the default-expand rule, it does NOT
   auto-expand (respect user's collapsed choice). Only the badge updates.

### 7.4 What Does NOT Update

To preserve the existing selective-refresh architecture:
- Artifact content that has been fetched and cached: stays in the cache
- User's expand/collapse choices: never overridden by polling
- Sidebar content (Plan Tasks, Completions, Traces): updated by existing
  data-dynamic logic, unchanged from current behavior

## 9. Accessibility

### 9.1 Semantic HTML Structure

The step explorer uses a `<section>` wrapper with proper heading hierarchy.
Each stage is a `<section>` with an `<h3>` in the header (the page h1 is the
slug, the sidebar cards use h2, so stages are h3).

```html
<section class="step-explorer" aria-label="Pipeline stages">
  <div class="step-explorer-controls">
    <h2 class="step-explorer-title">Step Explorer</h2>
    <button class="step-explorer-expand-all" aria-label="Expand all stages">
      Expand All
    </button>
    <button class="step-explorer-collapse-all" aria-label="Collapse all stages">
      Collapse All
    </button>
  </div>

  <div class="stage" data-stage="intake" data-status="done">
    <button class="stage-header"
            aria-expanded="false"
            aria-controls="stage-body-intake"
            id="stage-header-intake">
      <span class="stage-step-number" aria-hidden="true">1</span>
      <h3 class="stage-name">Intake</h3>
      <span class="badge stage-status-done" aria-label="Status: done">Done</span>
      <time class="stage-timestamp" datetime="2026-03-28T14:32:00Z">
        Mar 28, 14:32
      </time>
      <span class="stage-chevron" aria-hidden="true"></span>
    </button>
    <div class="stage-body"
         id="stage-body-intake"
         role="region"
         aria-labelledby="stage-header-intake"
         hidden>
      <!-- artifact list -->
    </div>
  </div>

  <!-- more stages... -->
</section>
```

### 9.2 ARIA Attributes

| Element | Attribute | Value | Purpose |
|---|---|---|---|
| Step explorer wrapper | `aria-label` | "Pipeline stages" | Identifies the landmark |
| Stage header button | `aria-expanded` | "true" / "false" | Communicates expand state |
| Stage header button | `aria-controls` | stage body id | Links header to content |
| Stage body | `role` | "region" | Identifies collapsible region |
| Stage body | `aria-labelledby` | stage header id | Labels region by header |
| Stage body | `hidden` | (present / absent) | Hides content from all users |
| Status badge | `aria-label` | "Status: done" etc. | Readable status for SR |
| Step number | `aria-hidden` | "true" | Decorative, SR reads stage name |
| Chevron | `aria-hidden` | "true" | Decorative indicator |
| Expand All button | `aria-label` | "Expand all stages" | Clear action label |
| Collapse All button | `aria-label` | "Collapse all stages" | Clear action label |
| Artifact View button | `aria-expanded` | "true" / "false" | Content viewer state |
| Artifact View button | `aria-controls` | viewer id | Links to content panel |
| Loading indicator | `aria-live` | "polite" | Announces load completion |
| Loading indicator | `role` | "status" | Semantic loading status |

### 9.3 Keyboard Navigation

The step explorer supports full keyboard navigation:

| Key | Context | Action |
|---|---|---|
| Tab | Anywhere | Move focus to next focusable element (stage headers, artifact View buttons, Expand/Collapse All) |
| Enter / Space | Stage header focused | Toggle expand/collapse |
| Enter / Space | Artifact View button | Toggle content viewer |
| Enter / Space | Expand All / Collapse All | Execute action |
| Escape | Content viewer open | Close the viewer, return focus to View button |
| Arrow Down | Stage header focused | Move focus to next stage header |
| Arrow Up | Stage header focused | Move focus to previous stage header |
| Home | Stage header focused | Move focus to first stage header |
| End | Stage header focused | Move focus to last stage header |

The stage headers form a logical group. Arrow key navigation within the group
follows the WAI-ARIA accordion pattern. Tab moves focus out of the accordion
to the next element on the page.

Implementation: Each stage header is a `<button>` element (natively focusable
and activatable). The stage body uses `hidden` attribute (not `display: none`
with CSS) so it is removed from both visual and accessibility trees when
collapsed.

### 9.4 Screen Reader Announcements

When a stage is expanded:
- Screen reader announces: "[Stage Name], expanded" (via aria-expanded)
- If content is loading: "Loading artifacts..." (via aria-live on loader)
- When load completes: "3 artifacts loaded" (via aria-live on loader region)

When a stage transitions status via polling:
- The sr-status region (existing from task 72) announces:
  "Execution stage is now in progress" or "Planning stage completed"

### 9.5 Color Contrast

All color combinations meet WCAG AA (4.5:1 for text, 3:1 for UI components):

| Element | Foreground | Background | Ratio |
|---|---|---|---|
| Stage name on default bg | #1a1a2e | #ffffff | 16.2:1 |
| Status "done" badge | #065f46 | #d1fae5 | 5.4:1 |
| Status "in_progress" badge | #92400e | #fef3c7 | 5.8:1 |
| Status "not_started" badge | #6b7280 | #f3f4f6 | 4.6:1 |
| Timestamp text | #6b7280 | #ffffff | 4.6:1 |
| Muted not_started header | #1a1a2e at 0.6 | #ffffff | ~8:1 |

### 9.6 Reduced Motion

All animations respect `prefers-reduced-motion: reduce`:

```css
@media (prefers-reduced-motion: reduce) {
    .stage-chevron,
    .stage-body,
    .artifact-viewer,
    .stage-status-badge,
    .artifact-loading-bar::after {
        transition: none;
        animation: none;
    }
}
```

When reduced motion is preferred:
- Chevron snaps to rotated position instead of animating
- Stage body appears/disappears instantly (no slide)
- Loading bar shows static indeterminate state
- Status badge transitions happen instantly

## 10. User Flows

### 10.1 First Visit to an Active Item

1. User navigates to `/item/{slug}`
2. Page renders with step explorer; Execution stage is "in_progress"
3. Execution stage is auto-expanded (default expand behavior)
4. User sees 2 artifacts loaded so far (task outputs)
5. "Loading..." indicator appears briefly for each artifact content
6. Content renders; user reads the execution output
7. 10 seconds later: polling updates. A new artifact appears in the list
   with a subtle slide-in animation
8. User clicks on Intake stage header to see the original request
9. Intake expands; Execution stays expanded (multi-expand)
10. Both stages' expand states are saved to localStorage

### 10.2 Return Visit to a Completed Item

1. User navigates to a previously viewed item (now completed)
2. localStorage has saved expand states from last visit
3. Stages restore their saved states
4. All stages show "done" with completion timestamps
5. No polling active (item is completed)
6. User clicks "Expand All" to review the full pipeline history

### 10.3 Watching a Pipeline Progress in Real Time

1. User opens a freshly queued item
2. All stages are "not_started" except Intake which is "in_progress"
3. Intake is auto-expanded (active stage rule)
4. Polling detects Intake transitions to "done"
5. Badge animates to green "Done"; timestamp appears
6. Requirements stage badge transitions to "In Progress" with pulse dot
7. User clicks Requirements to expand it
8. Requirements artifacts appear; user reads them
9. Pattern continues through Planning, Execution, Verification, Archive
10. On final transition to "completed", polling stops

### 10.4 Screen Reader User Exploring Stages

1. SR user navigates to the page
2. Reads header information, reaches "Step Explorer" heading
3. Tabs to first stage header: "Intake, button, expanded"
4. SR reads the region contents: "3 artifacts"
5. Tabs through artifact View buttons
6. Presses Enter on "View" for User Request
7. "Loading artifacts..." announced
8. "Content loaded" announced; content is readable
9. Presses Escape to close viewer, focus returns to View button
10. Arrow Down to move to Requirements stage header
11. Enter to expand it; continues exploring

### 10.5 Keyboard-Only User

1. User tabs through page, reaches step explorer
2. Tab lands on first stage header (Intake)
3. Arrow Down cycles through stage headers (Requirements, Planning, etc.)
4. Enter on Execution stage to expand it
5. Tab into the stage body to reach artifact View buttons
6. Enter on View button; content loads inline
7. Tab through content (scrollable via keyboard)
8. Escape closes viewer; focus returns to the View button
9. Tab out of stage body to next stage header
10. Shift+Tab moves backwards

## 12. Edge Cases

### 12.1 Empty Stages

A stage with "done" status but zero artifacts (unlikely but possible):
- Stage header shows "Done" badge and timestamp
- Expanding shows: "No artifacts recorded for this stage." (muted text)

### 12.2 Verification Stage Omitted for Features

Feature items do not have a Verification stage. The stage is simply not
rendered in the explorer. The step numbers adjust accordingly (1-5 instead
of 1-6). The omitted stage does not leave a gap or "hidden" placeholder.

### 12.3 Very Long Artifact Content

Content viewer has max-height: 400px with overflow-y: auto. This prevents
a single large artifact from pushing other stages off-screen. The user can
scroll within the viewer.

### 12.4 Many Artifacts in Execution Stage

The Execution stage can have dozens of per-task output files. These are
rendered as a scrollable list within the stage body (no max-height on the
artifact list itself -- only on individual viewers). The user can scroll
through all artifacts.

### 12.5 All Stages Not Started (New Item)

When all stages are "not_started":
- Intake stage is expanded by default (even though empty)
- Shows: "No artifacts yet. Waiting for pipeline to start."
- Other stages are muted and not clickable

## 14. Design Rationale Summary

### Why multi-expand accordion instead of single-panel?
Users need to compare artifacts across stages (e.g., checking structured
requirements against the design document). A single-panel accordion would
force constant toggling between stages. Multi-expand lets the user keep two
or more stages open side by side.

### Why button elements for stage headers instead of details/summary?
The `<details>/<summary>` pattern is simpler but lacks control over animation,
keyboard navigation (arrow keys), and programmatic state management (localStorage
persistence, default expand logic). Using `<button>` with `aria-expanded` and
`hidden` gives full control while maintaining accessibility parity.

### Why not auto-expand stages on live status transitions?
When a stage transitions to "in_progress", auto-expanding it would override
the user's deliberate choice to keep it collapsed. This is disruptive,
especially if the user is reading another stage. The status badge update is
sufficient to signal the transition; the user can expand when ready.

### Why relative timestamps instead of absolute?
Relative timestamps ("3h ago", "12m ago") are more immediately meaningful
when monitoring active pipeline runs. Users care about "how long ago" more
than the exact time. Absolute timestamps are used for older entries (>6 days)
where the relative format becomes less useful.

### Why a loading bar instead of a spinner?
The loading indicator sits inside a narrow horizontal artifact viewer. A
spinner would require vertical space and feels like a modal blocker. A
horizontal progress bar fits the layout naturally, conveys forward progress,
and disappears cleanly when content loads.

### Why not lazy-load stage headers too?
Stage headers (name, status, timestamp) are lightweight metadata (a few
hundred bytes total for all 6 stages). Loading them on demand would add
a flash of empty content and complicate the initial page render. Only the
heavy content (artifact file contents) is lazy-loaded.
