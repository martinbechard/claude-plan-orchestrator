## 1. Design Philosophy

The item detail page must keep users informed of processing progress without
disrupting their reading flow. The current full-page meta refresh violates this
by resetting all DOM state every 10 seconds. The fix must be invisible to the user
when they are reading static content, and clearly visible when dynamic values change.

Three principles:

1. **Imperceptible refresh** -- The user should never notice the refresh cycle
   unless they are actively watching a dynamic value change. No flash, no scroll
   jump, no layout shift, no collapsed sections.
2. **Trust the DOM** -- Static content sections are rendered once by the server
   and never touched again. The JS refresh logic has no reason to interact with
   static containers, so they are structurally excluded from updates.
3. **Graceful degradation** -- If JavaScript is disabled or fetch fails, the page
   still shows the last server-rendered state. No broken UI, no error overlays.
   The user simply sees a static snapshot.

## 2. Dynamic vs Static Element Map

### 2.1 Dynamic Elements (updated every 10 seconds)

These elements contain values that change during pipeline execution. Each is
identified by a `data-dynamic` attribute on the innermost element holding the
value text.

| Element | data-dynamic value | Update strategy | Parent container |
|---|---|---|---|
| Pipeline stage badge | `stage-badge` | Replace entire badge HTML (class + text) | `.item-badges` |
| Active worker banner | `worker-banner` | Replace entire banner innerHTML or hide | Standalone `div` above layout |
| Cost value | `cost` | Replace textContent | `.item-header-stats` |
| Token count | `tokens` | Replace textContent | `.item-header-stats` |
| Duration value | `duration` | Replace textContent | `.item-header-stats` |
| Velocity stat | `velocity` | Replace textContent (value + unit) | `.item-header-stats` |
| Velocity badge | `velocity-badge` | Replace textContent or show/hide | `.item-badges` |
| Plan task list | `plan-tasks` | Replace innerHTML of `ul.task-list` | Plan Tasks card |
| Plan task count | `plan-task-count` | Replace textContent ("3 / 8") | Plan Tasks card header |
| Validation results | `validation-results` | Replace innerHTML of results container | Validation Results card |

### 2.2 Static Elements (never touched by refresh)

These elements are rendered once by the server and are never modified by
the JS refresh logic. They contain no `data-dynamic` attributes.

| Element | Why static |
|---|---|
| Breadcrumb nav | Slug and navigation path never change |
| Item slug heading (h1) | The slug is the page identity |
| Item type badge | Item type is fixed at creation |
| Requirements card (full rendered HTML) | Requirements markdown does not change during execution |
| Original request disclosure | Raw backlog content is fixed |
| Output artifacts card | Artifacts are discovered at page load; new ones appear on next full navigation |
| Completion history table | Historical completions are immutable; new ones appear on next full navigation |
| Traces table | Traces are recorded externally; appears on next full navigation |
| Console output section | Log files are loaded on-demand via existing fetch-on-toggle |
| Last trace link in header | Updated with velocity badge, but the link target is static during a run |

### 2.3 Boundary Rules

The key design invariant: **the JS update function receives a flat JSON object
and uses `document.querySelector('[data-dynamic="X"]')` to locate each target
element.** Any element without a `data-dynamic` attribute is invisible to the
refresh logic. This makes the boundary self-documenting in the HTML template.

No element that contains a `<details>` or `<summary>` tag should ever receive
a `data-dynamic` attribute, because replacing its innerHTML would reset the
open/closed state. The one exception is the plan task list, which contains no
user-expandable disclosures.

## 3. Interaction Preservation

### 3.1 details/summary State

The item page has several disclosure widgets:

| Disclosure | Location | User action to open |
|---|---|---|
| Original request toggle | Inside requirements card | Click "Original request" |
| Verification evidence | Inside completion history rows | Click "Evidence" |
| Verification findings | Inside completion history rows | Click verdict badge |
| Console output section | Below two-column layout | Click "Console Output" |
| Console output previews | Per log file entry | Click "Preview" |
| Artifact inline viewers | Per artifact entry | Click artifact path |

**All of these are inside static containers.** The refresh logic never touches
the completion history table, console output section, or requirements card.
Therefore, all disclosure open/closed states are preserved automatically -- no
explicit state tracking or restoration is needed.

### 3.2 Scroll Position

Because the refresh updates only textContent or small innerHTML fragments of
specific elements, the browser scroll position is undisturbed. There is no
reflow that would cause a scroll jump.

The only potential layout shift is the active worker banner appearing or
disappearing. This is addressed in Section 4.2.

### 3.3 Text Selection

If the user is selecting text within a dynamic element (e.g., copying the cost
value), the refresh will replace the textContent and disrupt the selection. This
is acceptable because:
- Dynamic values are short (a few characters)
- The selection use case is rare for these elements
- The alternative (diffing text nodes) adds complexity for negligible benefit

Static content selection (requirements, original request, etc.) is never affected.

## 4. Refresh Indicator UX

### 4.1 No Visible Loading Indicator

The refresh should be invisible. Rationale:

- The fetch is lightweight (~1 KB JSON) and completes in <100ms on localhost
- A spinner or pulse animation would draw attention to the refresh cycle,
  which is the opposite of the design goal (imperceptible updates)
- The existing pulsing dot on the active worker banner already signals that
  the item is being processed -- that is sufficient visual feedback
- Adding a refresh indicator would create a "flickering dashboard" feel that
  makes users anxious about whether the page is working correctly

**Decision: No loading spinner, no progress bar, no visual indicator of the
fetch cycle.** The values simply update in place, like a live dashboard.

### 4.2 Active Worker Banner Transitions

The worker banner appears when a worker is active and disappears when it finishes.
This is the one dynamic element that changes the page layout (it takes vertical space).

**Appear behavior:**
- When the JSON response contains `active_worker` data but the banner is currently
  hidden, insert the banner HTML. Use a CSS transition (`max-height` + `opacity`,
  200ms ease) to slide it in smoothly rather than popping in abruptly.

**Disappear behavior:**
- When the JSON response has `active_worker: null` but the banner is currently
  visible, animate it out (same transition in reverse) then remove the DOM element.

**Respects `prefers-reduced-motion`:** When reduced motion is preferred, skip the
animation and show/hide instantly.

### 4.3 Value Change Highlight (Optional Enhancement)

A subtle highlight effect when a dynamic value changes would help users notice
updates without being distracting:

- When a `data-dynamic` element's new value differs from its current textContent,
  apply a brief CSS transition: `background-color` from `#fef3c7` (warm yellow)
  to `transparent` over 600ms.
- This creates a gentle "flash" on the changed value, similar to financial
  dashboards showing price changes.

**This is an optional enhancement.** If implementation complexity is a concern,
skip it entirely. The core requirement is met without it.

## 5. Graceful Degradation

### 5.1 JavaScript Disabled

When JavaScript is disabled:
- The page renders normally via the server-side template (Jinja2)
- No meta refresh tag is present (it has been removed)
- The user sees a static snapshot of the item at the time of navigation
- All content is fully readable and all disclosures work
- The page is still useful -- the user can simply refresh manually (F5)

**No fallback meta refresh.** Adding a `<noscript>` meta refresh tag would
reintroduce the exact problem we are solving. The user who has disabled JS
has accepted that live updates won't work.

### 5.2 Fetch Failure (Network Error or Server Error)

When a fetch cycle fails:

1. **Silently retry on the next interval.** Do not show an error message to the
   user for a single failed fetch. The page still shows the last successfully
   rendered state.

2. **After 3 consecutive failures**, show a small, non-intrusive status message
   below the header card:
   ```
   [i] Live updates paused -- unable to reach server. Retrying...
   ```
   - Style: muted text (13px, color #6b7280), left-aligned, no background color
   - Not an alert or error banner -- it's informational
   - Disappears immediately when a fetch succeeds

3. **After 10 consecutive failures**, stop the interval timer and update the
   status message:
   ```
   [i] Live updates stopped. Reload the page to resume.
   ```
   - Adds a "Reload" link that calls `location.reload()`
   - No further fetch attempts until the user reloads

### 5.3 Server Returns Non-JSON or 404

If the endpoint returns an unexpected content type or a 404 (e.g., item was
deleted while user was viewing it):

- Treat it as a fetch failure (same counter and retry logic as 5.2)
- After the 3-failure threshold, the informational message appears
- The page remains usable with the last-rendered content

### 5.4 Partial JSON Response

If the JSON response is valid but missing expected fields:

- Update only the fields that are present in the response
- Skip fields that are missing (do not blank out the current value)
- This makes the update function forward-compatible with endpoint changes

## 6. Accessibility of Live-Updating Content

### 6.1 aria-live Regions

Dynamic elements that update silently need to be announced to screen reader
users at an appropriate frequency. Using `aria-live` correctly is critical to
avoid overwhelming the user.

| Element | aria-live | aria-atomic | Rationale |
|---|---|---|---|
| Active worker banner container | `polite` | `true` | Already has `role="status"` in current template. Announces full banner text when worker appears/disappears/changes task. |
| Pipeline stage badge | `off` | -- | Stage changes are rare (a few times per item). Announce via the worker banner instead, which provides richer context. |
| Cost / Tokens / Duration / Velocity | `off` | -- | These change every 10 seconds. Announcing them would be extremely noisy. Users can query these values on demand via standard screen reader navigation. |
| Plan task list | `off` | -- | Task status changes are low-frequency but the list is long. Announcing every task update would be disruptive. |
| Validation results | `polite` | `true` | New validation results are significant events. Announce when a new result appears. |
| Error status message (Section 5.2) | `polite` | `true` | Connection problems should be announced. |

**Key decision: Most dynamic values are `aria-live="off"`.** The user can navigate
to them at any time to read the current value, but they are not pushed to the
screen reader automatically. Only state-change events (worker appears/disappears,
validation results arrive, connection errors) warrant live announcements.

### 6.2 Semantic Wrapper for Dynamic Content

Add a visually hidden `<div>` with `aria-live="polite"` and `role="status"` that
receives a text summary of significant state changes:

```html
<div class="sr-only" aria-live="polite" role="status"
     data-dynamic="sr-status"></div>
```

The JS update function writes a text string into this element when:
- A worker starts: "Worker started on task #0.1 Systems design"
- A worker finishes: "Worker finished. Pipeline stage is now completed."
- A validation result arrives: "New validation result: PASS for task 1.1"

This provides a curated, infrequent stream of announcements that gives screen
reader users awareness of progress without the noise of every metric update.

### 6.3 Existing Accessibility Preserved

The current template already has good accessibility patterns that the selective
refresh must preserve:

- `aria-label` on stat containers ("Total API cost", "Total tokens", etc.)
- `role="status"` and `aria-live="polite"` on the active worker banner
- `aria-label` on the plan task list
- `aria-hidden="true"` on decorative separators

The JS update function must preserve these attributes when replacing innerHTML.
For elements where only `textContent` is replaced, attributes are naturally
preserved. For elements where `innerHTML` is replaced (worker banner, plan
tasks, validation results), the replacement HTML must include the same
accessibility attributes as the server-rendered version.

### 6.4 Focus Management

If the user has focus on a dynamic element when it updates:
- `textContent` updates do not move focus (browser behavior)
- `innerHTML` updates may destroy the focused element if it is inside the
  replaced container

**Mitigation for innerHTML updates:**
- Before replacing innerHTML of the plan task list, check if any element inside
  it has focus. If so, record the focused element's position and restore focus
  after the update.
- The worker banner and validation results are not keyboard-focusable (they
  contain no interactive elements that the user would tab to), so focus loss
  is not a concern for those containers.

## 7. Timing and Lifecycle

### 7.1 Refresh Interval: 10 Seconds

Maintain the same 10-second interval as the current meta refresh. This is
familiar to users who have already observed the refresh cycle.

### 7.2 Start Condition

The refresh timer starts only when the server-rendered pipeline stage indicates
the item is active (not "completed" and not "unknown"). This is communicated
via a data attribute on the page:

```html
<div data-refresh-active="true" data-refresh-url="/item/{{ slug }}/dynamic"
     data-refresh-interval="10000"></div>
```

The JS reads these attributes on page load. If `data-refresh-active` is not
`"true"`, no timer is started.

### 7.3 Stop Condition

When the JSON response contains `pipeline_stage` equal to "completed" or
"unknown", the JS calls `clearInterval` to stop the timer. The page becomes
fully static from that point.

### 7.4 Tab Visibility

When the browser tab is backgrounded (`document.visibilityState === "hidden"`):
- Pause the refresh timer to avoid unnecessary network requests
- Resume when the tab becomes visible again
- Perform an immediate fetch on resume to catch up on any changes

This saves server resources for users who open the page and switch to another
tab while waiting.

## 10. User Flows

### 10.1 Happy Path: Watching an Item Process

1. User navigates to `/item/{slug}`
2. Page renders with current state (server-side)
3. User expands the "Original request" disclosure to read it
4. 10 seconds later: cost, tokens, duration update silently
5. The "Original request" disclosure remains open
6. User scrolls down to read requirements
7. Another refresh cycle: plan task status updates (a task changes from
   "pending" to "in_progress")
8. User does not notice the update unless they look at the plan task card
9. Worker finishes; banner animates out; stage badge updates to "completed"
10. Refresh timer stops. Page is now fully static.

### 10.2 Expanding and Reading While Active

1. User arrives while item is being processed
2. User opens verification evidence disclosure in completion history
3. Multiple refresh cycles pass
4. Completion history is in the static zone -- evidence stays open
5. User finishes reading, collapses the disclosure manually
6. No disruption occurred

### 10.3 Network Interruption

1. User is watching an active item
2. Network drops (e.g., laptop sleeps, VPN reconnects)
3. Fetch fails silently for cycle 1, 2
4. Cycle 3 fails: "Live updates paused -- unable to reach server. Retrying..."
   appears below header
5. Network recovers at cycle 5: fetch succeeds, message disappears, values update
6. User may not have noticed anything if they were reading requirements

### 10.4 Screen Reader User

1. Screen reader user navigates to item page
2. Reads through header stats via standard navigation
3. Active worker banner announces via aria-live when worker starts a new task
4. User continues reading requirements (static, uninterrupted)
5. When worker finishes, sr-status region announces "Worker finished."
6. User navigates back to header to check final cost/tokens

## 11. AC Traceability

| AC | How this UX design addresses it |
|---|---|
| AC1 | details/summary elements are all in static containers; JS never touches them (Section 3.1) |

| AC3 | Static sections have no data-dynamic attributes; update function cannot reach them (Section 2.3) |
| AC4 | Refresh timer stops on "completed"/"unknown" pipeline_stage; runs only while active (Section 7.2, 7.3) |

| AC7 | Only data-dynamic elements updated; all other DOM untouched including disclosures (Section 2.3, 3.1) |
| AC8 | No full-page reload means browser never resets details/summary state (Section 3.1) |

## 12. Design Rationale Summary

### Why no loading indicator?
A spinner or pulse animation on every 10-second cycle would make the page feel
like it is constantly loading. The goal is the opposite: the page should feel
stable and calm, with values updating so smoothly that the refresh is imperceptible.
The active worker banner's pulsing dot already signals ongoing processing.

### Why aria-live="off" for most dynamic values?
Announcing cost, tokens, duration, and velocity every 10 seconds would make the
page unusable for screen reader users. Only significant state changes (worker
start/stop, validation results, connection errors) warrant live announcements.
Users can query current values on demand by navigating to them.

### Why no noscript fallback meta refresh?
A noscript meta refresh would reintroduce the exact problem (collapsing sections)
for users without JavaScript. It is better for those users to have a static page
they can manually refresh than to have the same broken experience as before.

### Why innerHTML replacement for plan tasks instead of surgical DOM updates?
The plan task list can grow, shrink, and change order during execution. Diffing
individual task items and patching them would be complex and fragile. Replacing
the entire `<ul>` innerHTML is simpler and the plan task list contains no
user-expandable disclosures, so no interaction state is lost.

### Why pause refresh on hidden tabs?
Background tabs waste server bandwidth when nobody is looking at them. The Page
Visibility API is well-supported and trivial to implement. Resuming with an
immediate fetch on tab-visible ensures the user sees current data when they
switch back.
