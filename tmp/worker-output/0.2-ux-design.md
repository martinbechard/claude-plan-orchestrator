## 1. Design Philosophy

The execution history UI serves a single core user need: **understanding what the
pipeline did for a specific work item, at any level of detail**. The design follows
three principles:

1. **Progressive disclosure** -- Show high-level summary first, let users drill
   down only when they choose to. Never overwhelm with all details at once.
2. **Context preservation** -- When drilling into a subtree, never lose sight of
   where you are in the overall execution. Breadcrumbs and visual hierarchy
   maintain spatial orientation.
3. **Zero dead ends** -- Every piece of data the user might want is reachable.
   No depth cutoffs, no hidden information, no "see raw JSON for more."

## 2. User Personas and Goals

### Primary persona: Pipeline operator
- Monitors completed work items for quality
- Needs to quickly see: Did it succeed? How long? How much did it cost?
- Drills into failures to understand what went wrong

### Secondary persona: Developer/debugger
- Investigating unexpected agent behavior
- Needs to see exact prompts sent, responses received, tool call sequences
- Uses the deep-dive view heavily

### Tertiary persona: Cost analyst
- Reviews execution costs across phases
- Needs accurate aggregated costs at every level
- Compares cost across items via the completions table

## 3. Information Architecture

### 3.1 Page Hierarchy

```
Completions Page (entry point)
  |
  +-- [Trace link per row] --> Execution History Page
                                  |
                                  +-- Header: item slug, status, duration, cost
                                  |
                                  +-- Tree Panel (left, 40%)
                                  |     |
                                  |     +-- Root node (pipeline run)
                                  |     |     +-- Phase nodes (intake, planning, ...)
                                  |     |           +-- Agent nodes
                                  |     |                 +-- Tool call nodes
                                  |     |                       +-- Sub-tool calls (recursive)
                                  |     |
                                  |     +-- [Expand/collapse at every level]
                                  |
                                  +-- Detail Panel (right, 60%)
                                  |     |
                                  |     +-- Node summary (type-specific content)
                                  |     +-- Metrics bar (latency, tokens, cost, model)
                                  |     +-- Observability metadata (structured)
                                  |     +-- Raw JSON toggle
                                  |
                                  +-- [Deep-dive button] --> Deep-Dive Modal
                                        |
                                        +-- System Prompt panel (left)
                                        +-- Agent Response panel (right)
                                        +-- Metrics footer
```

### 3.2 Information Density by Level

| Tree Level | Visible in tree row | Detail panel shows |
|---|---|---|
| Root (pipeline run) | Item slug, overall status badge, total duration, total cost | Summary stats, phase breakdown table, worker logs |
| Phase (intake, planning, ...) | Phase name, phase number, status badge, duration, cost | Agent name, activity summary pills, files read/written, bash commands, artifacts |
| Agent (Claude CLI session) | Agent name/type, status, duration, cost | Prompt preview (truncated), response preview, model name, token count |
| Tool call (Read, Edit, Bash...) | Tool name, icon, truncated target (file path or command) | Full input (file path, command, search pattern), full output/result, latency |
| Sub-tool call | Same as tool call | Same as tool call |

## 4. User Flows

### 4.1 UC1: Navigate from Completions to Execution History

**Flow:**
1. User lands on Completions page (existing)
2. Sees table with Slug, Trace, Type, Outcome, Cost, Duration, Velocity, Finished At
3. Clicks "Trace" link in the Trace column for a specific row
4. Browser navigates to /execution-history/<run_id>
5. Execution History page loads with the tree pre-expanded to phase level

**Design details:**
- The "Trace" column link uses the existing blue link style (#3b6fcf)
- Link text shows truncated run_id (first 8 chars) for developer reference
- Rows without a run_id show em-dash instead of link
- The standalone "Traces" nav link in base.html is removed
- Back navigation from Execution History returns to Completions (breadcrumb)

**Entry point change:** The Completions page becomes the SOLE way to reach execution
history. The nav bar "Traces" link is removed. This simplifies the mental model --
users think "I want to see what happened for item X" and go to Completions to find it.

### 4.2 UC2: Explore the Recursive Execution Tree

**Flow:**
1. User arrives at Execution History page
2. Tree panel (left side) shows root node with phases as immediate children
3. Phase nodes are initially collapsed (showing name, status badge, duration, cost)
4. User clicks a phase node -- it expands to show agent children
5. User clicks an agent -- it expands to show tool call children
6. User can continue expanding indefinitely (no depth limit)
7. Clicking an expanded node collapses it (toggle behavior)

**Design details:**
- Tree uses standard disclosure triangle (chevron) affordance
- Indentation: 20px per level, with vertical connector lines
- Each tree node row height: 32px (compact but touch-friendly)
- Node row structure: [chevron] [icon] [name] [status badge] [duration] [cost]
- Icons by node type:
  - Phase: numbered circle (01, 02, ...) in phase color
  - Agent: robot icon (or agent glyph)
  - Tool call: tool-specific icon (file for Read, pencil for Edit, terminal for Bash)
  - Default: circle dot
- Keyboard navigation: Arrow keys to move between nodes, Enter/Space to expand/collapse
- Currently selected node has a highlighted background (#eff4ff)
- Tree is scrollable independently of the detail panel

**Expand/collapse interaction:**
- Single click on chevron or node row: toggle expand/collapse
- Single click on node row (when already open): also selects node for detail panel
- "Expand All" / "Collapse All" buttons at top of tree panel for batch operations
- Animated expansion (150ms slide-down) for spatial continuity

### 4.3 UC3: View Node Details in Side Panel

**Flow:**
1. User clicks any node in the tree (or it is auto-selected on page load)
2. Detail panel (right side) updates to show content specific to that node type
3. Panel content scrolls independently of the tree panel

**Panel content by node type:**

#### Graph/Phase nodes
- **Header:** Phase name + status badge
- **Metrics bar:** Duration | Cost | (model if available)
- **Content:** Activity summary as pills (Read 5 files, edited 2, ran 8 bash)
- **Files section:** Files read list, Files written list (monospace, truncated with hover tooltip)
- **Bash commands:** List of commands run (truncated at 120 chars)
- **Artifacts:** Design docs, plan files, logs referenced
- **Observability:** Validator verdicts (PASS/FAIL with details), pipeline decisions, exit codes

#### Agent nodes
- **Header:** Agent name + model badge + status badge
- **Metrics bar:** Latency | Tokens (input/output) | Cost | Model
- **Prompt tab:** Truncated system prompt preview (first 500 chars)
- **Response tab:** Truncated agent response preview (first 500 chars)
- **Deep-dive button:** Opens the full prompt/response deep-dive modal
- **Observability:** Plan state snapshot if available

#### Tool call nodes
- **Header:** Tool name + icon + status
- **Metrics bar:** Latency (when available)
- **Input section:** Full tool input formatted by type
  - Read: file_path, offset, limit
  - Edit: file_path, old_string preview, new_string preview
  - Write: file_path, content length
  - Bash: command (full, in monospace block)
  - Grep: pattern, path, output_mode
  - Glob: pattern, path
- **Output section:** Tool result (truncated at 2000 chars with "show more" toggle)
- **Observability:** Subprocess exit codes for Bash calls

**Conditional metrics display (FR8):**
- Each metric in the metrics bar is only shown when the data exists
- No placeholder values, no "N/A" -- simply omit the metric
- If no metrics are available, the metrics bar is hidden entirely

**Raw JSON toggle (FR7):**
- Small "Show raw data" toggle button in the panel footer
- Initially hidden (toggle off)
- When toggled on: shows collapsible JSON blocks for inputs_json, outputs_json, metadata_json
- JSON is syntax-highlighted (keys in blue, strings in green, numbers in orange)
- Toggle can be turned off to re-hide

### 4.4 UC4: Deep-Dive Prompt/Response Inspection

**Flow:**
1. User selects an agent node in the tree
2. Detail panel shows agent summary with "Deep Dive" button
3. User clicks "Deep Dive" button
4. Modal overlay appears with two side-by-side panels
5. Left panel: System Prompt (full text, scrollable)
6. Right panel: Agent Response (full text, scrollable)
7. Metrics footer below panels: Latency, Token count (input + output)
8. User can scroll each panel independently
9. User closes modal via X button, Escape key, or clicking overlay backdrop

**Design details:**
- Modal takes 90% of viewport width, 85% of viewport height
- Each panel has a header label ("System Prompt" / "Agent Response")
- Each panel has a copy button (clipboard icon) in its header
- Panel backgrounds use dark theme (#1a1a2e) with light text for code readability
  (matching the prototype screens)
- Text is monospace, 13px, with comfortable line-height (1.5)
- Panels have a 50/50 split with a 1px divider
- If only one side has content (e.g., no system prompt), that panel shows
  "No system prompt recorded" in muted text
- Modal backdrop has slight blur (backdrop-filter: blur(2px)) and 50% opacity overlay
- Focus is trapped within the modal while open (accessibility requirement)
- On close, focus returns to the "Deep Dive" button that triggered the modal

### 5.2 Responsive Behavior

- **>= 1024px:** Side-by-side tree + detail panels (40/60 split)
- **768px - 1023px:** Tree panel collapses to 35%, detail panel 65%
- **< 768px:** Stacked layout -- tree on top (max-height: 40vh, scrollable),
  detail panel below. Tree selection scrolls detail into view.

### 5.3 Split-Pane Interaction

- No resizable divider (fixed proportions to avoid complexity)
- Both panels scroll independently (overflow-y: auto)
- Tree panel has min-width: 280px to prevent node truncation
- Detail panel has min-width: 360px for readable content

## 6. Interaction Patterns

### 6.1 Tree Node Selection

| Action | Result |
|---|---|
| Click node row | Select node (show in detail panel) AND toggle expand/collapse |
| Click chevron only | Toggle expand/collapse WITHOUT changing selection |
| Arrow Up/Down | Move selection to prev/next visible node |
| Arrow Right | Expand selected node (if collapsed) |
| Arrow Left | Collapse selected node (if expanded), or move to parent |
| Enter / Space | Toggle expand/collapse of selected node |
| Home / End | Jump to first / last visible node |

### 6.2 Page Load Behavior

1. Tree loads with root node and phase-level children visible (depth 1 expanded)
2. First phase node is auto-selected, its details shown in panel
3. Tree data is fetched in one request (full recursive tree from API)
4. If the tree has > 200 nodes, a small count badge shows "247 spans" in the header

### 6.3 Loading States

- **Initial page load:** Skeleton shimmer in both tree and detail panels
- **Tree expansion:** Instant (all data already loaded; no lazy loading needed
  since the recursive CTE fetches everything)
- **Deep-dive modal:** Instant open (data already available from the selected node)

### 6.4 Empty States

- **No tree data:** "No execution spans found for this run. The trace may still
  be recording." with a link back to Completions.
- **No detail for selected node:** "Select a node in the tree to view its details."
  (only shown when page first loads before auto-selection fires)

## 9. Accessibility (WCAG 2.1 AA)

### 9.1 Semantic Structure

- Tree panel uses `role="tree"` with `role="treeitem"` for each node
- Each treeitem has `aria-expanded="true|false"` for expand state
- Each treeitem has `aria-level` matching its depth (1-based)
- Each treeitem has `aria-selected` for the active selection
- Detail panel uses `role="region"` with `aria-label="Node details"`
- Deep-dive modal uses `role="dialog"` with `aria-modal="true"` and `aria-labelledby`

### 9.2 Keyboard Navigation

- Tree is fully keyboard-navigable (see Section 6.1)
- Tab order: Tree panel -> Detail panel -> Detail panel controls
- Deep-dive modal traps focus (Tab cycles within modal)
- Escape closes the deep-dive modal
- All interactive elements are focusable and have visible focus indicators
- Focus indicator: 2px solid #3b6fcf outline with 2px offset

### 9.3 Screen Reader Support

- Tree nodes announce: "[name], [status], [depth] of [total levels], [expanded/collapsed]"
- Detail panel changes announce via `aria-live="polite"` region
- Status badges have `role="status"` (existing pattern)
- Icons are decorative (`aria-hidden="true"`) with text labels always present

### 9.4 Color and Contrast

- All text meets 4.5:1 contrast ratio against its background
- Status badges use both color AND text labels (never color-only)
- Phase colors are supplemented by numbered indicators (01, 02...)
- Tree connector lines are supplemental (not the only depth indicator -- indentation
  and aria-level provide the primary structure)

### 9.5 Motion and Reduced Motion

- Expand/collapse animations respect `prefers-reduced-motion: reduce`
- When reduced motion is preferred: instant show/hide (no slide animation)
- Modal backdrop fade-in also respects reduced motion

## 12. Design Rationale Summary

### Why split-pane instead of inline expansion?
The tree + detail panel pattern (used in Chrome DevTools, VS Code explorer, etc.)
scales to arbitrary depth without layout thrashing. Inline expansion of detail
within tree nodes would create jarring height changes and make deep trees
disorienting. The fixed detail panel provides a stable reading area.

### Why a modal for deep-dive instead of a new page?
The deep-dive view is a focused inspection of a single agent's prompt/response.
A modal preserves context (the tree and detail panel remain behind it) and
supports quick back-and-forth between multiple agent nodes. Users can close the
modal, select a different agent, and re-open. A new page would lose tree state.

### Why load all tree data at once?
The recursive CTE fetches the full subtree in a single query. Typical execution
trees have 50-300 nodes -- well within browser memory limits. Lazy loading would
add complexity for marginal benefit and would introduce loading delays at every
expansion click, degrading the feeling of instant exploration.

### Why remove the Traces nav link?
The Traces page duplicates functionality that now lives on the Completions page.
Removing it simplifies navigation and prevents the "which page do I use?" confusion.
Users have one mental model: Completions page shows all items, click Trace to drill in.
